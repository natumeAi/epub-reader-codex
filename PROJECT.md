# EPUB Reader Project

## 项目目标

做一个自托管的 EPUB 阅读网页应用，主要供个人使用，最终部署到飞牛 NAS。

核心目标：

- 支持上传、管理和阅读 EPUB 电子书。
- UI 尽量接近 Apple Books 的观感与交互气质。
- 以手机竖屏为唯一主要使用场景，移动优先设计。
- 书架支持自由拖动排序。
- 书架支持将一本书拖到另一本书中间位置时创建文件夹，交互参考多看阅读。
- 文件夹内部也支持与书架一致的拖动排序。
- 不支持文件夹嵌套文件夹。
- 阅读器使用 epub.js，实现基础阅读、进度保存和后续阅读体验增强。
- 通过 Docker 部署到飞牛 NAS，目标硬件为 x86 J3455 CPU、8G 内存。

## 技术栈

前端：

- React
- Vite

- 移动优先 CSS

后端：

- 飞牛 NAS 自托管部署

## UI 和产品约束

- 以竖屏窄视口 UI 为主要体验目标，不优先考虑桌面端、平板横屏或响应式大屏体验。
- 页面宽度、触摸区域、拖动交互、底部导航和阅读器控制都按竖屏窄视口设计。
- 支持手机、平板竖屏，以及桌面浏览器手动缩窄成竖屏窗口使用；在宽屏设备上内容区域应限制最大宽度并居中，保持与手机竖屏接近的观感，不扩展为桌面管理界面。
- 书架封面长宽保持固定，不随视口拉伸；书架根据可用宽度自动增减每行书籍数量，例如手机竖屏约 3 本一行，平板竖屏可显示更多列，宽屏仍限制最大内容宽度并居中。
- 书架界面展示封面和名称：书籍封面下方显示书名，文件夹封面下方显示文件夹名；名称最多显示两行，超出省略，避免挤压拖拽命中区域。
- 书架页面需要有更接近真实书架的背景氛围，可以使用克制的层板、浅木色或轻量拟物阴影，但整体仍保持 Apple Books 式简洁，不做厚重装饰。
- 书架顶部显示“继续阅读”区域，展示最近读过的几本书，按阅读进度更新时间倒序排列，最近读的排在最前；该区域不参与书架拖拽排序，不改变用户手动整理的根书架顺序。
- 风格参考 Apple Books：简洁、克制、轻量拟物感、重视封面展示和阅读沉浸感。
- 避免复杂后台管理界面，优先做个人使用时最顺手的核心流程。
- 所有拖动操作都要适合触摸屏使用。

## 书架与文件夹规则

- 书架根层使用统一排序列表，列表项可以是书籍或文件夹；文件夹在书架上像书籍一样参与拖动排序。
- 一本书同一时间只能处于一个位置：要么在书架根层，要么在某个文件夹内，不支持同一本书同时出现在多个文件夹。
- 拖动创建文件夹时，文件夹默认命名为“新建文件夹”，后续通过重命名功能修改。
- 拖动 A 到 B 的封面中间创建文件夹时，A 作为文件夹内第一本书，B 作为第二本书。
- 书架拖拽使用意图识别：靠近封面时优先锁定目标并避免目标提前被挤走；拖到书籍封面中心区才创建文件夹；拖到真实间隙区才显示排序挤开。
- 本阶段只支持“书籍拖到书籍中间创建新文件夹”；书籍拖入已有文件夹、文件夹拖到文件夹、文件夹拖到书籍中间都不触发创建或嵌套。
- 文件夹只允许包含书籍，不允许包含文件夹；文件夹嵌套必须在数据模型和接口层禁止。
- 空文件夹自动删除；该规则在实现“从文件夹移出书籍”时同步落地。
- 后续支持将根书架书籍拖入已有文件夹，并支持将文件夹内书籍拖回根书架；该能力必须和文件夹打开界面、文件夹内排序、从文件夹移出能力配套实现，避免误拖后无法恢复。

## 编码约定

为了减少后续编码时的 token 消耗，并尽量避免跨对话失去方向、盲目堆代码、技术债累积等问题，开发时遵守以下规则。

### 全局方向

- 每个新对话先读 `PROJECT.md`，再决定下一步。
- 开始编码前先确认当前任务属于哪个阶段，不临时扩张目标。
- 只围绕当前清单项做最小必要改动，避免顺手重构无关模块。
- 遇到不确定需求时，先在 `PROJECT.md` 追加或修正约定，再写代码。
- 如果发现实现方向和项目目标冲突，优先停下来调整方案，而不是继续补丁式堆代码。

### Token 节省

- 回答和代码说明尽量短，只保留决策、变更点、验证结果。
- 不重复粘贴大段已有代码，优先引用文件路径和函数名。
- 修改文件前先用 `rg`、目录列表或局部读取定位上下文，避免整仓盲读。
- 单次任务只打开和修改必要文件，避免把无关文件带入上下文。
- 优先小步提交式开发：一个功能点、少量文件、可验证结果。
- 代码命名清晰，减少靠长注释解释逻辑。

### 实现原则

- 优先完成可运行的最小闭环，再逐步增强体验。
- 前后端接口先保持简单稳定，避免过早设计复杂抽象。
- 能用成熟库解决的核心问题不要手写，例如 EPUB 解析、拖拽排序、触摸交互基础能力。
- 数据模型必须服务书架、文件夹、排序和阅读进度四个核心场景。
- UI 组件按手机竖屏真实使用来拆分，不为桌面端提前增加复杂度。
- 文件夹只允许包含书籍，不允许包含文件夹。

### 技术债控制

- 每实现一个功能，必须同步更新“开发进度清单”。
- 临时方案必须在代码或 `PROJECT.md` 里标记原因和后续处理方式。
- 不在同一轮里同时做功能、重构、换技术方案，除非这是完成任务所必需。
- 重复代码出现第三次前再考虑抽象，避免为了“优雅”提前抽象。
- 数据库结构变更要通过迁移文件表达，不直接依赖手工改库。
- Docker 和 NAS 部署相关配置要保持简单，优先稳定运行和数据可恢复。

### 验证规则

- 每次改动后至少做一种验证：构建、测试、接口调用或页面运行检查。
- 书架拖拽、文件夹拖拽、阅读器交互必须用手机竖屏视口验证。
- 涉及数据写入时，要验证重启后数据仍然存在。
- 涉及 Docker 部署时，要验证持久化目录不会被镜像重建覆盖。

## 阅读器控制层约定

阅读时的调整能力集中在阅读器控制层，通过中间点击呼出/隐藏。

- 顶栏：返回、书名、进度%。
- 底栏两个入口：
  - **目录**：独立面板，列章节，点击跳转。
  - **Aa 设置**：一个面板集中放阅读设置；字体大小和字体选择放入“字体”二级菜单，页边距、行距、字距、主题保留在主设置层。
- 设置改动即时作用于 epub.js（`rendition.themes` / `flow` 等），不落桌面式复杂设置页。
- 阅读器固定使用平移翻页，默认开启且不提供用户切换；动画只做 `transform` 横向滑动，不做 opacity 淡入淡出；真实翻页只由 epub.js `next/prev` 执行一次，不直接改 epub.js 内部滚动位置。
- 目录、Aa 面板均从底部滑起，与控制层显隐一致的触摸交互。

## PWA 约定

PWA 只用于改善手机主屏幕入口和独立窗口打开体验，不作为离线阅读或缓存方案。

- 实现范围仅包括 Web App Manifest、应用名称、图标、主题色、启动地址、竖屏方向和 iOS 添加到主屏幕相关 meta。
- 不实现离线阅读。
- 不注册 Service Worker。
- 不缓存 EPUB 文件、封面、书籍列表接口、阅读进度接口或其他后端数据接口。
- 应用内容始终以当前 Docker 服务提供的数据为准，避免缓存导致书架、文件夹和阅读进度状态过期。
