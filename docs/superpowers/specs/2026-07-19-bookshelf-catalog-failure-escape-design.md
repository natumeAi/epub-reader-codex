# Bookshelf Catalog Failure Escape Design

## Status

Approved in conversation on 2026-07-19.

## Goal

修复 2A–2D 整体审查确认的 P1：目录首次成功后，如果用户处于“最近添加”或“文件夹”等派生视图，后续目录刷新失败或持续加载时，用户仍能返回“全部 + 手动顺序”的可编辑书架。

## Constraints

- 只修复目录不可用状态下无法退出派生视图的问题。
- 保持所有依赖目录数据的视图与排序控件禁用。
- 不使用旧目录数据恢复其他控件，不自动切换当前视图。
- 不修改目录加载、错误展示、搜索或书架编辑行为。
- 不处理整体审查记录的 P2 及以下 backlog。

## Root Cause

`LibraryHome` 在目录加载中、加载失败或尚未加载时，将 `catalogControlsDisabled` 设为 true。`LibraryViewToolbar` 把该状态无差别应用到所有视图按钮和排序选择框。因此用户一旦停留在依赖目录数据的派生视图，目录刷新失败或挂起会同时禁用“全部”按钮，切断返回基础书架的路径。

## Chosen Design

`LibraryViewToolbar` 在 `controlsDisabled` 为 true 时，仅继续禁用非 `LIBRARY_VIEW.ALL` 的视图按钮：

```jsx
disabled={controlsDisabled && option.value !== LIBRARY_VIEW.ALL}
```

排序选择框维持禁用，其他目录派生视图按钮也维持禁用。“全部”按钮沿用 `useLibraryView.selectView(LIBRARY_VIEW.ALL)` 的现有行为，清空搜索并恢复手动顺序；不引入新的状态、effect 或回退数据。

该行为既提供故障逃生路径，也不允许用户继续操作依赖当前目录的控件。系统不会在刷新失败时自动离开用户当前视图；只有用户主动点击“全部”才切换。

## Alternatives Rejected

- **目录异常时自动切回“全部”**：会在暂时加载或失败时主动丢弃用户的视图状态，行为更具侵入性。
- **使用旧目录保持全部控件可用**：会让依赖目录数据的操作基于可能过期的数据，违背当前禁用策略。

## Testing Strategy

按一个 RED/GREEN 循环增加集成回归测试：

1. 先以可用目录进入“最近添加”派生视图。
2. 重新渲染目录错误状态，断言“全部”仍可用，而“最近添加”、其他目录视图和排序选择框仍禁用。
3. 点击“全部”，断言恢复“手动顺序”和可编辑书架。
4. 使用同一测试覆盖目录加载状态的控件可用性，确保刷新挂起时也保留同一逃生路径。

先运行该定向测试并确认因“全部”按钮被禁用而失败；完成最小实现后只重新运行原定向验证命令。最终按实施计划指定的定向构建命令验证。

## Success Criteria

- 目录错误或加载状态下，“全部”按钮保持可用。
- 目录依赖视图按钮与排序选择框继续禁用。
- 用户点击“全部”后恢复“全部 + 手动顺序”的可编辑书架。
- 不自动切换视图，不读取旧目录恢复控件，不改变其他行为。
- 新增回归测试和计划指定的定向构建命令通过。
